## Why This Was Done

Phase 1.1 introduced a major architectural change: the web UI became fully read-only and all article write surfaces (create, edit, delete) were removed. Claude manages content directly via the database repository layer.

This raised two questions that unit tests alone could not answer:

1. **Do the pages actually render correctly?** — The article list, detail page, and dashboard all depend on live DB queries. A unit test with mocked data would not catch a broken DB connection or a mis-rendered page.

2. **Is the read-only constraint actually enforced?** — Removing routes from code does not guarantee they are gone from the running server. An E2E test that sends a real HTTP `POST` to `/api/articles` confirms this at the network level.

The solution was to set up Playwright E2E tests that run against the real Next.js dev server and local PostgreSQL database.

Full rationale for choosing Playwright: [ADR-014](../decisions/014-e2e-testing-playwright.md)

## What Was Set Up

### Infrastructure
- Installed `@playwright/test` as a dev dependency
- Installed Chromium browser binary via `playwright install chromium`
- Created `playwright.config.ts` with:
  - Chromium as the test browser
  - `webServer` config to auto-start Next.js dev server
  - `baseURL` set to `http://localhost:3000`
  - `DATABASE_URL` injected for local PostgreSQL

### Test Files

| File | What It Covers |
|------|---------------|
| `e2e/dashboard.spec.ts` | Stat cards visible, no write buttons present |
| `e2e/articles.spec.ts` | Article list renders, empty state, 404 for unknown slug |
| `e2e/layout.spec.ts` | Sidebar nav links, header title, theme toggle, API read-only |

### Scripts Added to `package.json`

```bash
npm run test:e2e        # Run all 17 E2E tests headlessly
npm run test:e2e:ui     # Run with Playwright's interactive UI
```

### Skill Created

`/dawnbase-test-e2e` — Claude skill that runs the full E2E suite and reports results, with failure diagnosis.
